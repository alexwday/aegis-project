4. Model Structure and Approach

Model Name: IRIS (Intelligent Research and Information System)
Model ID: IRIS-001

4.1 Approach Selection

IRIS is a qualitative model that leverages large language models (LLMs) within a Retrieval-Augmented Generation (RAG) framework. The model's underlying theory combines natural language processing with information retrieval to provide accurate finance and accounting guidance. This approach was selected because it constrains LLM outputs to information found in authorized documentation, ensuring responses are grounded in RBC Finance policies rather than potentially inaccurate model-generated content.

The model selection process was driven by available models within RBC's infrastructure, particularly through RBC's Lumina portal. After reviewing industry practices and benchmarks, we selected GPT-4o for complex reasoning tasks and final response generation, GPT-4o-mini for efficient processing of routine decisions, and text-embedding-3-large for generating vector representations in semantic search. For specialized processing needs, we employ Qwen2 7B VL for visual content and Cohere command-north-large for external data.

IRIS implements a modular, agent-based architecture with two processing paths:

  • Direct Response Path: For queries answerable using only conversation history
  • Research Path: For complex queries requiring database research, following a full pipeline (Router → Clarifier → Planner → Database Subagents → Summarizer)

The system employs five document processing methods tailored to different document types:

  • Catalog Search - Small: For standard documents, converting content to markdown with document-level metadata
  • Catalog Search - Vision: For visual documents, using vision models to extract text and visual elements
  • Catalog Search - Excel: For spreadsheets, preserving table formatting and structure
  • Catalog Search - Large: For large documents with sections, creating both document and section-level summaries
  • Semantic Search - Large: For reference documents, employing chunking, vector embeddings, and sophisticated reranking

We evaluated several alternative approaches before selecting our hybrid architecture:

  • Basic RAG: Rejected because it would query all databases for every question, reducing efficiency
  • Pure LLM Approach: Rejected because it could provide information not backed by official policy documentation
  • Single-Stage Processing: Rejected because it would limit strategic decision-making about information retrieval

Our selected approach aligns with IRIS's purpose of providing accurate, policy-compliant guidance based on authorized knowledge sources while maintaining transparency and efficiency. We chose a qualitative rather than quantitative model because our primary goal is to retrieve and synthesize finance and accounting policy information from documented sources, requiring natural language understanding and contextual reasoning rather than numerical predictions or classifications. There are no plans to transition to a quantitative model as the nature of policy guidance is inherently qualitative.

4.2 Model Inputs

IRIS uses three key inputs in its processing pipeline:

Conversation History: The complete record of user-system interactions, including the current query and all previous exchanges. This provides essential context for understanding the current query and enables continuity in multi-turn conversations. The conversation history is particularly critical for follow-up questions or references to previously discussed topics.

PostgreSQL Database Content: The system connects to structured database tables (as defined in the postgres_schema.sql file) that store processed document content and metadata. These tables form the knowledge base from which relevant information is retrieved during the research process. Table structure is optimized for different retrieval methods, with specialized fields supporting both catalog-based and vector-based search approaches.

Specialized System Prompts: Throughout the processing pipeline, carefully crafted prompts guide each LLM component. These prompts define the role and expected output format for each agent and vary based on the specific task (routing, clarification, planning, etc.). They effectively program the behavior of each component in the pipeline.

The agent architecture processes these inputs by strategically injecting prompts at each stage and incorporating relevant database content as context for the LLM. This approach allows the system to maintain conversation continuity while providing document-grounded responses to finance inquiries. Our input selection was driven by the need to balance comprehensive knowledge access with efficient processing. Alternative input approaches (such as direct document indexing without preprocessing) were considered but rejected because they would not support the specialized retrieval methods required for different document types.

Information about these limitations is communicated to users through the Maven interface and through disclaimers included in system responses. For each limitation, risk mitigants have been implemented:

Knowledge Boundaries: The system explicitly acknowledges when information is not found in available sources, providing clear statements about the limits of its knowledge. This prevents users from assuming the absence of information means the topic doesn't exist.

Context Window Limitation: The model implements sophisticated chunking techniques and section expansion methods to optimize information retrieval within context constraints. For very large documents, the system employs a recursive retrieval approach that progressively expands relevant sections.

Hallucination Risk: All responses include source citations and verification disclaimers. The system is designed to clearly indicate the source of information and to separate high-confidence information (from direct quotes) from synthesized content.

Response Latency: For complex research queries, the system provides progress updates during processing. The interface clearly indicates when research is in progress, maintaining user engagement during longer processing times.

There are no known situations that would produce unreasonable outputs beyond the identified limitations. The system operates within RBC's secure environment, with data retained for audit and monitoring purposes. No user query data is used for model training, maintaining privacy and security standards.
