5. Model Development Data

5.1 Data Validity and Sources

The IRIS system retrieves information from a collection of internal and external accounting and finance knowledge sources stored in a PostgreSQL database. This section describes the data used by the system, its sources, and processing methods.

IRIS accesses data covering several key domains. These include Internal RBC Policies, such as Corporate Accounting Policy Manuals (CAPMs), process documentation, and control frameworks. The scope also extends to Accounting Guidelines, encompassing RBC Finance-developed guides, cheatsheets, and wiki entries. Furthermore, Internal Documentation like accounting memos, project approval requests, and finance-related guidance is covered. The system also processes information related to Financial Reporting, including management reporting policies and external reporting requirements. Finally, it incorporates Accounting Standards, such as IFRS standards and interpretations from select accounting firms (EY, KPMG, PwC). This domain-focused coverage enables IRIS to address finance and accounting inquiries relevant to RBC Finance.

Data validity is assured through the source management approach. All data is provided directly by subject matter experts (SMEs) and database owners who upload the raw data files to designated Network Attached Storage (NAS) drive folders. Each database has a designated owner/SME responsible for maintaining the content and ensuring its accuracy. Since the data is directly uploaded by these authorized content owners, we assume the data is valid and authoritative. This approach ensures that the system's knowledge base contains only officially approved content from appropriate domain experts.

The system stores processed documents in a PostgreSQL database with a structured schema designed to support both catalog-based and semantic search approaches. The database schema is defined in the postgres_schema.sql file and includes tables for document metadata, content storage, vector embeddings, and performance monitoring. All data sources are obtained directly from the SME-maintained NAS folders, ensuring compliance with RBC's enterprise data management processes.

5.2 Data Cleaning and Processing

IRIS utilizes five specialized database refresh pipelines for transforming raw source documents into optimized formats for retrieval. These pipelines are tailored to specific document types and are activated as needed when SMEs upload new or updated content to the NAS drive input folders.

1. Catalog Search - Small Pipeline: This is the standard pipeline for processing normal documents like PDFs and DOCX files. It uses Azure Document Intelligence to convert content to markdown format while preserving essential structure. The pipeline processes entire documents as single units, generating document-level metadata. GPT-4o is then used to generate document usage summaries and descriptions, creating concise overviews of document purpose and applicability. This focuses on document-level metadata to support catalog selection and produces standardized metadata fields for consistent retrieval.

2. Catalog Search - Large Pipeline: This pipeline handles large documents with distinct sections, such as Corporate Accounting Policy Manuals. It also uses Azure Document Intelligence for initial conversion to markdown, but additionally implements section detection algorithms to split documents into logical sections based on headers. For these documents, GPT-4o generates both document-level and section-level summaries. A human-in-the-loop verification step is included to confirm proper section/chapter splitting is being performed correctly. This creates hierarchical metadata reflecting document structure and enables more granular selection by allowing LLMs to refine searches to specific sections.

3. Catalog Search - Excel Pipeline: Designed specifically for Excel spreadsheets containing tabular data, this pipeline uses Python ETL scripts to convert spreadsheet content to markdown while preserving table structures. It processes each relevant row or group of rows as a separate document entry and standardizes formatting for numerical data and table layouts. After conversion, GPT-4o generates document-level descriptions and usage summaries similar to the small pipeline.

4. Catalog Search - Vision Pipeline: This specialized pipeline processes infographics and visual content, particularly cheatsheets. It uses the Qwen2 vision model to process image content into structured markdown, extracting text, visual elements, and structural information. These results are combined into a single document with proper formatting, and then GPT-4o generates relevant metadata as in the small pipeline.

5. Semantic Search - Large Pipeline: This pipeline is used for very large reference materials such as IFRS Standards. It begins with Azure Document Intelligence to extract text while preserving document structure and identifies chapters and sections with hierarchical relationships. The pipeline creates optimally sized chunks for semantic search and then employs GPT to generate several layers of metadata:

  • Comprehensive metadata at the chapter level, including chapter summaries, applicable accounting standards, and key concepts
  • Section-level summaries that encapsulate the core content of each section
  • Importance scores for each section, indicating how critical that section's content is to the overall chapter topic
  • Extraction of key metadata and tags at the section level, including applicable standards, related concepts, and cross-references

This rich metadata is then applied at a post-chunk level to maintain context across the chunked data. Vector embeddings are generated using the text-embedding-3-large model configured for 2000 dimensions, and both content and metadata are stored to support sophisticated reranking and context expansion during retrieval.

The database refresh process occurs on a scheduled basis for each knowledge source, with frequency determined by the update cadence of the underlying documentation. Internal RBC Finance policy documents are typically refreshed quarterly or when significant updates occur, while external sources like accounting standards are refreshed following major updates to those standards.

5.3 System Inputs and Database Structure

IRIS operates primarily on conversation history and content retrieved from its database structure. The system processes user queries through a sequential pipeline, beginning with the complete conversation history as the initial input. The Agent Router analyzes this history to determine whether to use only conversation context (Direct Response path) or to retrieve information from databases (Research path).

For queries following the Research path, the system performs additional processing: evaluating context sufficiency through the Agent Clarifier, selecting relevant knowledge sources via the Agent Planner, retrieving information using specialized methods through Database Subagents, and synthesizing findings with the Agent Summarizer.

The system primarily processes three types of inputs: Conversation History containing complete user-system interaction records, PostgreSQL Database Tables providing structured document content and metadata, and Specialized Prompts containing function-oriented instructions for each agent component.

For semantic search capabilities, the system uses the text-embedding-3-large model configured to generate 2000-dimensional vectors (rather than its maximum 3072 dimensions) to fit PostgreSQL database limits. This creates efficient vector representations of document chunks that enable sophisticated similarity-based retrieval while working within database constraints.

5.4 Data Assumptions and Limitations

Despite IRIS's robust data processing framework, certain assumptions and limitations affect its operation. In cases where data requirements are not fully met, the system implements a gap analysis process to identify missing information and prioritize future data collection efforts. This process includes tracking common query topics that lack sufficient documentation, monitoring response quality metrics, and gathering user feedback on information completeness.

These limitations are actively managed through several risk mitigation strategies. For Content Currency, the system includes version information and last-update timestamps in responses to ensure users are aware of potential staleness. The Visual Content limitation is addressed by providing text descriptions of complex visuals and referring users to original documents when detailed visual information is critical. For Information Boundaries, the system explicitly acknowledges when information may be incomplete and suggests alternative resources when appropriate.

The IRIS system is designed for continuous improvement through regular documentation updates and processing enhancements. Documentation updates include scheduled refresh cycles to incorporate policy changes and expansion of the knowledge base to cover additional finance domains. Processing enhancements focus on improvements to visual content processing for complex financial diagrams and better preservation of document relationships and cross-references. The system's modular design allows for component-level improvements without requiring complete system redesign.
