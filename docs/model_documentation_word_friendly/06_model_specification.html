6. Model Specification

This section describes the IRIS model methodologies and processing components, focusing on the agent-based architecture, query processing mechanisms, and mathematical foundations of information retrieval and synthesis.

Agent Architecture

The IRIS system implements a modular, pipeline-based architecture consisting of specialized AI agents that process user queries through a structured workflow. The overall architecture contains two primary processing paths managed by a central model orchestration engine.

Processing Pathways

IRIS features two main processing pathways for handling different types of queries based on their complexity and information needs. The Direct Response Path is employed for queries that can be answered using only conversation history, making it ideal for contextual questions that don't require additional research. This streamlined path follows a simple processing sequence: agent_router → agent_direct_response, minimizing processing time and resource usage for straightforward inquiries. In contrast, the Research Path is utilized for more complex queries requiring database research. This comprehensive path implements a full retrieval-augmented generation approach with the sequence: agent_router → agent_clarifier → agent_planner → database_subagents → agent_summarizer. By routing queries through the appropriate pathway, the system optimizes both response speed and quality based on the specific information needed.

Agent Components and Functions

The IRIS agent architecture consists of specialized components, each performing a specific role in the query processing pipeline. The Router Agent serves as the initial decision point, analyzing incoming queries and conversation context to determine whether to process through the direct or research path. It employs a classification algorithm based on transformer embeddings with context-awareness to make accurate routing decisions. The Clarifier Agent evaluates whether there is sufficient context to proceed with research, requesting additional information when needed or formulating comprehensive research statements for clear queries. It uses information-theoretic query expansion techniques with carefully tuned relevance parameters to maximize information density while maintaining query focus.

The Planner Agent strategically selects the most appropriate databases for each query using multi-class relevance scoring that incorporates database-specific prior weightings based on source authority and content coverage. This ensures efficient resource utilization by targeting only the most relevant knowledge sources. Database Subagents implement five distinct processing methodologies (detailed in Section 6.2) for retrieving and pre-processing information from various data sources, each optimized for specific document types and structures. The Summarizer Agent integrates findings from multiple sources into coherent, well-structured responses using a multi-document extraction-abstraction mechanism with attention-based relevance weighting to prioritize the most important information. Finally, the Direct Response Agent generates answers using only conversation history when appropriate, employing constrained inference with optimized context-window utilization to produce accurate, contextually relevant responses without database access.

System Integration

The central chat model orchestration component (model.py) serves as the integration hub for the entire IRIS system. This component manages system initialization and configuration, ensuring all agents are properly loaded with their required settings and dependencies. It handles the critical function of query routing based on Router agent decisions, directing each query through either the Direct Response Path or the Research Path. The orchestration engine controls both sequential and parallel agent execution flow, allowing certain components to operate concurrently when appropriate while maintaining proper dependencies between stages. It also handles result aggregation and presentation, combining outputs from multiple agents into coherent, well-formatted responses for users. Additionally, the orchestration component incorporates comprehensive process monitoring and error handling, tracking performance metrics at each stage and managing exceptions to maintain system stability. This integrated approach ensures efficient coordination between specialized components while providing a unified experience to end users.

Database Retrieval Methodologies

The IRIS system implements five distinct database retrieval methodologies, each optimized for different types of source documents.

Catalog Search - Small

Mathematical Specification:

\begin{align} D_{selected} = LLM_{select}(catalog, q) \\ R_{synthesis} = LLM_{synthesize}(\{content(d) | d \in D_{selected}\}, q) \end{align}

Catalog Search - Small implements a two-stage document selection and synthesis process designed for standard documents with simple structures. The mathematical specification defines document selection as an LLM-based function that evaluates catalog metadata against the query to identify relevant documents. Once documents are selected, the response synthesis combines content from these documents to address the original query.

Implementation begins with catalog retrieval from PostgreSQL, querying document metadata to obtain names and descriptions. In the document selection phase, GPT-4o-mini (temperature 0.2) evaluates the catalog against the user query, selecting relevant documents based on their descriptions. Content retrieval then fetches the complete content for selected documents, and response synthesis uses GPT-4o to generate a comprehensive research report with proper citations.

This methodology handles various document types through specialized pre-processing pipelines. For tabular data (Excel files), a transformation process converts rows or logical groupings into document representations while preserving table structures in Markdown format. For visual content, a vision-to-text pipeline using the Qwen2 model extracts text, tables, charts, and diagrams from infographics and other image-based documents. After pre-processing, both Excel and vision-based documents follow the standard Catalog Search - Small retrieval pattern at query time. This approach is particularly well-suited for standard documents, spreadsheets with financial data, and visual resources like financial cheat sheets.

Catalog Search - Large

Mathematical Specification:

\begin{align} D_{selected} = LLM_{select}(catalog, q) \\ S_{selected,d} = LLM_{select}(sections(d), q) \text{ for each } d \in D_{selected} \\ R_{synthesis} = LLM_{synthesize}(\{content(s) | s \in \bigcup_{d \in D_{selected}} S_{selected,d}\}, q) \end{align}

Catalog Search - Large extends the standard catalog search with an additional layer of selection at the section level, making it ideal for large, structured documents. The mathematical specification outlines a three-stage hierarchical selection process that first identifies relevant documents, then selects appropriate sections within those documents, and finally synthesizes content from the selected sections. This approach provides more granular content selection compared to the Small methodology, enabling targeted information retrieval from extensive documents.

The implementation begins with the same catalog retrieval process used in the Small methodology, querying document metadata from the PostgreSQL database. Document selection follows a similar pattern, using GPT-4o-mini to evaluate document relevance based on catalog descriptions. What differentiates this methodology is the additional section summary retrieval step, which fetches section-level metadata for selected documents. This intermediate step enables the subsequent section selection process, where GPT-4o-mini evaluates each document's sections based on their summaries and selects only the most relevant ones for the query.

Content retrieval then becomes more targeted, fetching only the specific sections identified as relevant rather than entire documents. This selective approach maintains comprehensive coverage while significantly reducing the amount of text processed in the final synthesis stage. The response synthesis follows the same approach as in Catalog Search - Small, but operates on a more precisely curated set of content sections. This methodology is particularly effective for large documents with distinct sections or chapters, such as Corporate Accounting Policy Manuals and Global Finance Standards, where retrieving entire documents would be inefficient and potentially overwhelming.

Semantic Search - Large

Mathematical Specification:

\begin{align} E(c) = \vec{v_c} \\ E(q) = \vec{v_q} \\ sim(c, q) = 1 - (c.embedding <=> q.embedding) \\ C_{initial}(q) = top\_k(C, sim, k) \text{ where } k = 20 \\ C_{filtered} = \{c \in C_{initial} | LLM_{relevance}(c.summary, q) = 1\} \\ score(c) = sim(c, q) \cdot (1 + IMPORTANCE\_FACTOR \cdot c.importance\_score) \\ \text{where } IMPORTANCE\_FACTOR = 0.2 \end{align}

Semantic Search - Large implements a sophisticated vector-based similarity search with token-aware chunking, designed for processing extensive reference materials. The mathematical specification describes a multi-stage process that begins with embedding generation for document chunks and queries, creating dense vector representations that capture semantic meaning. Initial similarity scoring uses cosine similarity implemented through PostgreSQL's vector operator, identifying the most relevant chunks based on semantic closeness. A subsequent refinement stage applies LLM-based relevance filtering to remove irrelevant chunks despite high vector similarity. The importance reranking formula incorporates both vector similarity and predefined importance scores, prioritizing more significant sections with a balanced weighting factor. Additional specifications for section expansion and gap filling ensure comprehensive coverage with optimal context preservation.

The implementation begins with query embedding generation using the text-embedding-3-large model configured for 2000 dimensions, creating a dense vector representation of the user's query. This is followed by an initial vector similarity search that leverages PostgreSQL's vector extension to efficiently identify the top 20 most similar chunks based on cosine similarity. The system then applies summary relevance filtering, where GPT-4o-mini evaluates whether each chunk's summary is truly relevant to the query, removing irrelevant results while preserving the original ranking order.

The relevance-filtered chunks undergo importance-based reranking, which applies a scoring formula that combines vector similarity with section importance scores to prioritize more significant content. This is followed by section expansion based on token count, where top-ranked chunks with token counts below certain thresholds are expanded to include adjacent chunks from the same section, providing more comprehensive context. The system also implements sequence gap filling, which identifies and retrieves chunks that fill small gaps between consecutive chunks, creating a more continuous narrative. Finally, all selected chunks are formatted as "cards" with metadata and content, and GPT-4o generates a comprehensive research report with proper citations. This methodology is particularly effective for very large reference documents and external sources like accounting standards and professional guidance.

Model Decision Processes

IRIS implements four critical decision processes that guide query processing through the system pipeline.

Router Decision

The Router determines whether a query requires database research or can be answered directly from conversation history alone. This binary classification directs queries to either the Direct Response Path or the more comprehensive Research Path. Using GPT-4o-mini (temperature 0.1) for consistency, the Router analyzes the query and conversation context to evaluate whether existing information is sufficient for a response. The output is a function name to invoke ('response_from_conversation' or 'research_from_database'), optimizing resource usage by reserving the intensive research path only when necessary.

Clarifier Decision

The Clarifier evaluates whether there is sufficient context to proceed with research or if additional information is needed from the user. Operating on a sufficiency assessment scale, the system either requests clarification or formulates a comprehensive research statement when context is adequate. Using GPT-4o, the Clarifier produces either context questions (when information is insufficient) or a detailed research statement with specified scope ('metadata' for document discovery or 'research' for content analysis). This prevents wasted computational resources on vague queries while ensuring precise research direction.

Database Selection

The Planner determines which knowledge sources should be queried for a given research statement through relevance assessment. Using GPT-4o-mini (temperature 0.2), it analyzes the research requirements against available databases and selects only those exceeding a relevance threshold. This strategic selection is critical for system efficiency when dealing with multiple specialized knowledge sources. The output is a prioritized list of databases to query, focusing retrieval efforts on the most promising information sources while reducing processing overhead for each query.

Response Synthesis

The Summarizer integrates information from multiple database sources into a coherent, comprehensive answer. Taking detailed research from various subagents as input, GPT-4o (temperature 0.4) creates a cohesive synthesis that balances creativity with consistency. The output is a well-structured response that incorporates and properly cites all relevant sources while resolving conflicts and prioritizing authoritative content. This transformation of raw research results into user-friendly guidance ensures that users receive well-organized, properly attributed information that directly addresses their original query.

Model Parameters and Retrieval Specifications

The IRIS system utilizes specific model configurations and retrieval parameters tailored to optimize performance across different agent functions.

Language Model Parameters

IRIS strategically employs different language models with carefully tuned parameters for various system components. GPT-4o-mini serves as the workhorse for classification and decision-making tasks including Router Agent operations, Planner Agent database selection, Direct Response generation, and document/section selection processes. For these functions, the temperature is maintained between 0.1-0.2 to ensure low randomness and consistent decision-making, prioritizing reliability in these critical determination processes.

For more complex reasoning and content generation tasks, the system leverages GPT-4o with its enhanced capabilities. This model powers the Clarifier Agent's context assessment, the Summarizer Agent's content integration, and Database Subagent synthesis operations. The temperature settings for these tasks range from 0.3-0.7, with higher values applied to creative synthesis functions that benefit from some controlled variability. This balanced approach optimizes the trade-off between consistency and creative expression in generated content.

The system also incorporates Text-Embedding-3-Large for generating vector representations in the semantic search methodology. This embedding model is configured for 2,000 dimensions, providing high-dimensional, accurate similarity matching while remaining within the practical constraints of PostgreSQL's vector storage capabilities. This configuration offers an optimal balance between representational power and computational efficiency for similarity-based retrieval operations.

Retrieval Parameters

Each retrieval method in IRIS is configured with specific parameters optimized for its document type and search approach. For Catalog Search - Small, document selection employs LLM-based relevance assessment using GPT-4o-mini, with specific criteria that prioritize documents directly addressing the query. An explicit limit of 5 documents maximum is enforced per prompt instruction to manage context size effectively.

Catalog Search - Large implements a more sophisticated two-stage selection process that first identifies relevant documents and then selects appropriate sections within those documents. Selection guidance is provided through specific evaluation criteria embedded in prompts, with document selection limited to a maximum of 5 documents to ensure efficient processing.

The Semantic Search - Large methodology implements the most complex parameter structure. Its initial vector search applies document-specific thresholds tailored to different source types: IASB standards use separate thresholds (IAS: 20, IFRS: 20, IFRICS: 10, SIC: 10), while accounting firm sources each use a threshold of 20. The reranking method combines vector similarity with content importance (importance_factor=0.2), balancing semantic similarity with domain-specific significance. Final selection includes relevance filtering, section expansion (top 5 ranks ≤ 8000 tokens, others ≤ 4000 tokens), and sequence gap filling (gaps ≤ 8). The system implements dynamic token budget adjustment based on content relevance, typically 4,000-8,000 tokens.

Rather than using explicit numerical thresholds for most catalog-based methods, the system relies on comparative relevance assessment through carefully designed prompt engineering. These parameters have been empirically optimized through extensive testing to balance precision, recall, and processing efficiency. The result is a highly adaptable retrieval system that can handle diverse document types and query patterns while maintaining consistent performance standards.

Numerical Methods and Approximations

IRIS employs several key numerical techniques to optimize information processing and retrieval operations.

Tokenization

IRIS uses the tiktoken tokenizer (cl100k_base) for consistent token management across the system. Token thresholds are managed through dynamic truncation that preserves the most relevant information while respecting model limits. For chat history, the system keeps the last 10 messages within token limits, while document chunking targets sizes of 250-750 tokens per chunk with boundaries adjusted based on semantic considerations rather than strict token counts.

Token-Aware Chunking Algorithm

Mathematical Formulation:

\begin{align} \text{For document } D \text{ with sections } S = \{s_1, s_2, ..., s_n\} \\ \text{With thresholds } T_{min} = 250 \text{ and } T_{max} = 750 \\ \text{For each section } s_i:\text{ merge with adjacent sections if } T(s_i) < T_{min}, \\ \text{split if } T(s_i) > T_{max}, \text{ or keep as is if } T_{min} \leq T(s_i) \leq T_{max} \end{align}

The Token-Aware Chunking Algorithm segments documents by optimizing information density while preserving semantic structure. The system traverses document structures hierarchically to calculate token counts for each section using tiktoken. Small sections below the minimum threshold (250 tokens) are merged with adjacent sections, while large sections exceeding the maximum threshold (750 tokens) are split at natural paragraph boundaries. Sections already within the target range remain as standalone chunks. This approach balances processing efficiency with content coherence by respecting the document's logical flow while creating optimally sized chunks for embedding and retrieval operations.

Embedding Batch Processing

To optimize embedding generation, the system processes document chunks in batches of 50 items, submitting them to the text-embedding-3-large model as single requests. This parallelized approach significantly improves throughput by reducing API overhead while maintaining embedding quality. The resulting 2000-dimensional vectors are stored with their corresponding chunk IDs in PostgreSQL using the pgvector extension, enabling efficient similarity searches during query processing.

Similarity Scoring

Mathematical Formula:

\[sim(v_1, v_2) = \frac{v_1 \cdot v_2}{||v_1|| \cdot ||v_2||} \quad \text{where } sim \in [0,1]\]

For semantic search operations, IRIS employs cosine similarity to compare vector representations of document chunks and queries. This formula measures the cosine of the angle between two vectors, with values ranging from 0 to 1 for text embeddings (higher values indicating greater semantic similarity). Implemented through PostgreSQL's vector extension, this approach identifies conceptually related content regardless of exact keyword matching, enabling the system to capture meaningful relationships even when terminology differs between queries and documents.

Process Monitoring and Metrics

IRIS implements comprehensive process monitoring through a three-part approach that tracks performance metrics, stores historical data, and enables continuous optimization.

The metrics collection system captures key operational data across the processing pipeline, including stage-based timing metrics for each component, token usage statistics for cost analysis, API response times, and error conditions. These metrics provide visibility into all aspects of system performance, from router decisions to final response synthesis.

All collected metrics are stored in the PostgreSQL database's process_monitor_logs table, enabling historical analysis of performance patterns across different query types and usage patterns. This centralized storage facilitates the identification of bottlenecks, supports cost attribution through detailed token usage tracking, and enables system reliability monitoring through comprehensive error logging.

The process monitoring architecture creates a continuous feedback loop for system refinement by identifying processing bottlenecks, optimizing token usage patterns, correlating success/failure rates with specific query types, and fine-tuning model parameters based on empirical performance data. This data-driven approach ensures the system continuously improves in efficiency, accuracy, and cost-effectiveness over time.
