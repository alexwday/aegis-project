Iris Agent Framework

System Architecture Overview

1. Introduction

Iris is a framework designed for building modular and robust agentic applications. It provides a structured architecture for processing user queries through a sequence of specialized agents, enabling interaction with various data sources to generate informed responses. This document outlines the core architecture, workflow, and key capabilities of the Iris system.

2. Core Architecture & Workflow

Iris utilizes a component-based design where agents handle specific tasks in a defined workflow. The path taken depends on the query's requirements:

A. Workflow Paths

B. Core Components

The framework comprises the following key agents:

  • Router: Initial query analysis and path determination (Direct vs. Research). Located at agents/agent_router/.
  • Clarifier: Query refinement for the research path. Located at agents/agent_clarifier/.
  • Planner: Selects relevant data sources based on the clarified query. Located at agents/agent_planner/.
  • Database Router: Dispatches requests to appropriate Subagents. Located at agents/database_subagents/database_router.py.
  • Subagents: Interface with specific data sources (databases, APIs, RAG pipelines). Primary extension point. Located at agents/database_subagents//.
  • Summarizer: Synthesizes findings from Subagents into a final response. Located at agents/agent_summarizer/.
  • Direct Response Agent: Generates responses using only conversation history. Located at agents/agent_direct_response/.

3. Architectural Highlights

Key design principles and capabilities underpin the Iris framework:

A. Modular LLM Integration

Description: Interactions with Large Language Models (LLMs) are abstracted. Agents request functional capabilities (e.g., small, large, embedding) instead of hardcoding specific model names.

Implementation: A central connector (llm_connectors/) maps these capabilities to specific, configured LLM endpoints (defined in chat_model/model_settings.py). This allows easy swapping of LLM providers or models and facilitates cost/performance tuning by assigning different model tiers to different agent tasks.

B. Templated Data Source Connectivity

Description: A standardized pattern enables integration of diverse data sources through plug-and-play "Subagents".

Integration Process:

  1. Create a dedicated folder (agents/database_subagents/new_source/).
  2. Implement source-specific logic in subagent.py (using query_database_sync).
  3. Define LLM prompts and the output schema (SYNTHESIS_TOOL_SCHEMA) in a prompt file (e.g., content_synthesis_prompt.py).
  4. Register the source name in global_prompts/database_statement.py for discovery.

C. Structured Prompting Framework

Description: Systematic prompting techniques guide LLM behavior for consistency and control.

Key Techniques:

  • CO-STAR Framework: Defines Context, Objective, Style, Tone, Audience, and Response Format.
  • XML Tag Delimitation: Structures prompt sections (e.g., <CONTEXT>, <TASK>) for clarity.
  • Global Context Injection: Automatically includes shared information (project scope, data sources, constraints from global_prompts/).
  • Precise Tool Calling Schemas: Defines explicit structures (e.g., SYNTHESIS_TOOL_SCHEMA) for reliable, structured LLM tool outputs.

Example Prompt Structure (Router Agent)

This example shows the final prompt sent to the LLM for the Router Agent, including the dynamically injected global context (Project Statement, Fiscal Calendar, etc.) within the overall structure.

You are an expert routing agent in the IRIS workflow.

<CONTEXT>
  <!-- Dynamically Injected Global Context -->
  <PROJECT_STATEMENT>
    IRIS is an AI assistant designed to answer accounting and finance questions
    based on authoritative internal and external guidance documents.
  </PROJECT_STATEMENT>
  <FISCAL_CALENDAR>
    The current fiscal year is F2025, ending Oct 31, 2025.
    The current quarter is Q2 F2025 (Feb 1, 2025 - Apr 30, 2025).
  </FISCAL_CALENDAR>
  <DATABASE_STATEMENT>
    Available Databases:
    - internal_capm (Internal Control Policy)
    - internal_wiki (RBC Finance Wiki)
    - external_kpmg (KPMG Accounting Guidance)
    - external_pwc (PwC IFRS Manual)
    ... (List continues)
  </DATABASE_STATEMENT>
  <RESTRICTIONS_STATEMENT>
    CRITICAL_DATA_SOURCING: Only use information explicitly present in provided context.
    Never fabricate database content or guidance document information.
    ... (Other restrictions)
  </RESTRICTIONS_STATEMENT>
</CONTEXT>

<OBJECTIVE>
  To analyze each user query and determine the optimal processing path:
  1. Direct response from conversation context when sufficient information exists
  2. Research from databases when authoritative information is needed
</OBJECTIVE>

<STYLE>
  Analytical and decisive like an expert system architect.
  Focus on efficient, accurate classification of queries based on their information needs.
</STYLE>

<TONE>
  Neutral and objective.
  Focused solely on routing efficiency without emotional coloring.
</TONE>

<AUDIENCE>
  Internal system components that will process the query based on your routing decision.
  Your routing choice directly impacts the quality, authority, and efficiency of the final response.
</AUDIENCE>

<TASK>
  You are the initial step in the IRIS workflow, responsible for determining how to handle each user query.

  <ANALYSIS_INSTRUCTIONS>
    For each user query, analyze:
    1. The entire conversation history
    2. The latest question
    3. Whether sufficient information exists in the conversation
  </ANALYSIS_INSTRUCTIONS>

  <DECISION_CRITERIA>
    This system relies only on conversation context, not internal training data:
    - For accounting/finance topics, PREFER RESEARCH over direct response by default
    - Consider research especially when:
      * Query mentions specific accounting standards (IFRS, IAS, GAAP, etc.)
      * Query mentions specific financial reporting requirements
      * Query asks about implementations, interpretations, or applications of standards
      * Query mentions or implies the need for authoritative sources
      * Query contains database specificity phrases like "check guidance", "reference", etc.
      * Query asks to list, find, or search for items/documents/files within a specific database...
    - Only choose direct response when:
      * The question is extremely basic, definitional, or conceptual *and* does not require referencing specific database content...
      * ... (Other criteria)
  </DECISION_CRITERIA>

  <!-- Other sections like ROUTING_EXAMPLES, WORKFLOW_CONTEXT, IO_SPECIFICATIONS, ERROR_HANDLING omitted for brevity -->
</TASK>

<RESPONSE_FORMAT>
  Your response must be a tool call to route_query with exactly one of these function names:
  - "response_from_conversation"
  - "research_from_database"
  No additional text or explanation should be included.
</RESPONSE_FORMAT>

D. Operational Readiness Features

Description: Production-grade infrastructure components that enable reliable operation, monitoring, and maintenance of the system.

Key Components:

  • Process Monitoring Framework:
    • Built-in hooks (initial_setup/process_monitor.py) for comprehensive telemetry
    • Records execution time, token usage, and decision points at each step
    • Provides real-time visibility into system performance and bottlenecks
    • Enables detailed post-execution analysis of complete processing paths
  • Centralized Logging Architecture:
    • Standardized logging format and levels (initial_setup/logging_config.py)
    • Consistent error reporting and tracing across all components
    • Configurable output destinations (console, file, external services)
    • Supports both development debugging and production monitoring
  • Robust Conversation Management:
    • Flexible conversation state handling (conversation_setup/)
    • Supports different input message formats (OpenAI JSON, Claude, etc.)
    • Efficient management of lengthy conversation histories
    • Built-in truncation and allows for summarization of extended conversations
  • Reusable Security Components:
    • Enterprise-ready SSL configuration (initial_setup/ssl/)
    • OAuth-based authentication (initial_setup/oauth/)
